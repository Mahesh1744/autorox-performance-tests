Findings & Recommendations
✅
Zero downtime across all 6 test types
229,846 requests processed with only 1 transient failure (0.0004%). The API handled smoke, load, stress, spike, soak, and scalability scenarios without any service interruption.
✅
Dual-auth flow stable under all load levels
JSESSIONID + JWT authentication remained reliable across 100 concurrent VUs. Login latency (avg 287 ms) was consistent even during the 34-minute soak test with no session drops.
⚠
Response times spike 4.5x under stress load
p95 jumps from 329 ms (25 VUs) to 1,490 ms (100 VUs), indicating resource saturation around 60-80 VUs. Connection pool tuning or horizontal scaling is recommended before sustaining 100+ VU production traffic.
⚠
Slowest endpoints: paymentsReport & serviceList
/reports/paymentsReport (avg 634 ms, p95 1,780 ms) and /serviceAdvisor/serviceList (avg 523 ms, p95 1,420 ms) are the clear bottlenecks. Likely a missing DB index or N+1 query — run EXPLAIN ANALYZE on the underlying SQL.
✅
No memory leak in 34-minute soak test
Sustained load at 15 VUs showed a stable response time plateau (avg ~290 ms after minute 10) with no upward drift, confirming clean memory management and connection handling over extended periods.
✅
Fast spike recovery — high elasticity
After each 80 VU burst, response times returned to baseline within 30-60 seconds. The system is well-suited for handling sudden traffic surges such as morning workshop check-in rush hours.
💡
Recommendation: Cache the reports endpoints
getDashboardCount, workInProgressData, and paymentsReport are called frequently but change slowly. A 60-second server-side or Redis cache could cut avg response times by 40-60% at minimal implementation cost.
💡
Recommendation: Run a breakpoint test at 120-150 VUs
Stress test reached 100 VUs without failure. A breakpoint test incrementing by 10 VUs every 2 minutes will pinpoint the exact saturation threshold, critical for setting auto-scaling triggers and SLA limits.
