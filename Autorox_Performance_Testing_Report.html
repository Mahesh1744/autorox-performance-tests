2. Tools & Technologies Used

k6

Grafana k6 v2.0.0 — Performance Test Engine

Open-source load testing tool written in Go with a JavaScript scripting API. Used to write and execute all 7 test scenarios locally and in CI. Chosen for its low resource footprint, native JavaScript support, built-in metrics, and Grafana Cloud streaming capability.

GH

GitHub Actions — CI/CD Pipeline

Automated workflow (.github/workflows/k6-performance.yml) that runs all 7 test jobs on push/PR. Uses grafana/setup-k6-action@v1 for k6 installation. Includes regression gate (Python), Slack failure alerts, and artifact upload for each run.

Py

Python 3 — Regression Gate Script

check_regression.py parses p95 values from k6 log output and compares them against baseline.json. If the current run is more than 20% worse than the stored baseline, the CI job fails. Supports ms/s/µs/us unit parsing.

PS

PowerShell 5.1 — Local Automation

run_all.ps1 runs any single test or the full suite locally. update_baseline.ps1 extracts p95 from the latest log file and updates baseline.json automatically. All scripts are compatible with Windows PowerShell 5.1.

AI

Claude (Anthropic) — AI Development Assistant

Used throughout the project to architect the test suite, write all JavaScript test files, debug authentication flows, fix encoding issues, build the CI/CD workflow, create the regression gate, and execute test runs autonomously. See Section 3 for full details.

Git

Git / GitHub Desktop — Version Control

All test scripts, baselines, and CI configuration are version-controlled at github.com/Mahesh1744/autorox-performance-tests. Baseline JSON is committed after each stable test run to lock in reference values.

4. Authentication Architecture

The Autorox API uses a two-layer authentication system that required careful implementation in k6:

Step 1 — Spring Security Form Login

POST /axprprod/login
Content-Type: application/x-www-form-urlencoded

ssoId=<username>&password=<password>&csrf_token=dummy

Response: HTTP 302 + Set-Cookie: JSESSIONID=...

Step 2 — JWT Extraction

GET /axprprod/userAccess
Cookie: JSESSIONID=...
redirects: 0  <-- critical: must prevent auto-follow

Response: HTTP 302
Location: /axprprod/index?token=<base64url-encoded-JWT>

Step 3 — JWT Decode

// Extract token from Location header
const token = locationHeader.split('token=')[1];

// Decode using k6/encoding
import { b64decode } from 'k6/encoding';
const payload = JSON.parse(b64decode(parts[1], 'rawurl', 's'));
const workshopId = payload.workshop_id;  // e.g. 47282
Important note on field names The login form uses ssoId as the username field name (not username). This is specific to Autorox's Spring Security configuration and must be used exactly.

Token Usage in Tests

The JWT is returned from setup() and passed to all VU iterations as data.token. Every API request includes:

headers: {
  'Authorization': 'Bearer ' + token,
  'Content-Type': 'application/json',
}

Regression Gate Logic

The check_regression.py script parses the p95 value from the k6 log file using a regex, loads the stored baseline from baseline.json, and compares them:

if current_p95 > baseline_p95 * (1 + tolerance/100):
    print(f"REGRESSION: {current_p95}ms is {pct:.1f}% worse than baseline {baseline_p95}ms")
    sys.exit(1)  # fails the CI job

8. Key Findings & Recommendations

Finding 1 — Healthy at Normal Load

The soak test (150 VUs, 60 min) showed 0.00% error rate and p95 = 3.18s with all 259,061 checks passing. The server is stable and leak-free at production-level concurrent usage. No memory or connection pool degradation over 1 hour.

Finding 2 — Capacity Limit Around 250–300 VUs

The load test at 250 VUs produced p95 = 4s with 0.09% errors. The scalability test confirms degradation begins between 300–500 VUs. Recommended max concurrent load for pre-prod: 250 VUs before latency becomes user-noticeable.

Finding 3 — Server Cannot Handle Traffic Spikes

The spike test (instant 800 VU bursts) produced 26.47% error rate and p95 = 11.3s. The server has no auto-scaling or request queuing for sudden bursts. Any marketing campaign, push notification blast, or mass event that drives sudden traffic will cause significant errors.

Finding 4 — Breaking Point Is ~500 VUs Sustained

Both stress and scalability tests confirm the server breaks (error rate > 10%) around 500 VUs sustained. The breakpoint test should be run to identify the exact VU count where errors first exceed 10% for 30 seconds continuously.

Finding 5 — Tickets Endpoints Are Most Vulnerable

Under load, the first endpoints to time out are getServiceTicketsByStatus (OPEN and CLOSED) and workInProgressData. These likely involve expensive database queries without adequate indexing or caching.

Recommendations

  1. Add database query caching for the tickets and WIP data endpoints — these are read-heavy and good candidates for a Redis or in-memory cache with a 30–60 second TTL.
  2. Implement connection pooling review — the server's behavior above 500 VUs suggests the database connection pool may be exhausting. Review the pool size settings in the Spring Boot datasource configuration.
  3. Add rate limiting / request queuing to prevent the spike scenario from causing cascading failures. A queue-based approach would absorb bursts gracefully instead of returning errors.
  4. Scale up pre-prod hardware to better represent production capacity, or document that pre-prod is deliberately under-specced and adjust test thresholds accordingly.
  5. Run the breakpoint test to find the precise VU count where the system fails for 30 consecutive seconds — this is the hard limit that should never be reached in production.
  6. Set up Slack webhook (SLACK_WEBHOOK_URL GitHub secret) so the QA team gets immediate notification when CI performance tests fail.
